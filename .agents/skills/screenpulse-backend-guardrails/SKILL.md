# ScreenPulse Backend Skill

## 目标

修改后端时，优先保持现有架构，完成当前任务，不主动重写项目。

不要假设固定文件名。  
先识别当前项目的后端入口、路由、鉴权、数据模型、配置和业务服务所在位置，再修改。

---

## 基本原则

- 不为了“更高级”主动引入新框架、新数据库、新队列、新工作流系统。
- 不主动重构目录结构。
- 不把一个小任务扩大成架构迁移。
- 优先沿用项目已有风格、命名、响应格式和错误处理方式。
- 新增逻辑要放在最接近现有职责的位置。
- 不绕过已有鉴权、权限、配置和日志机制。

---

## 分层要求

保持职责清楚：

- 接口层只负责请求处理、参数校验、当前用户获取、权限判断和调用业务逻辑。
- 业务层负责截图处理、模型调用、摘要刷新、审计日志等副作用。
- 数据模型层只定义持久化结构。
- Schema / DTO 层只定义请求和响应结构。
- 配置层只管理环境变量和运行参数。

注意：

- 业务层不要反向依赖接口层。
- 接口层不要堆大量模型调用和文件处理细节。
- 不要把配置、密钥、模型地址硬编码在业务代码里。

---

## 鉴权要求

所有非公开接口都必须鉴权。

当前用户必须来自后端鉴权结果，例如 token、session 或项目已有 auth dependency。

禁止相信前端传来的用户身份字段。

错误做法：

```text
使用 request.user_id 当作当前用户
使用 body.user_id 当作当前用户
使用 query.user_id 当作当前用户
````

正确做法：

```text
从后端鉴权机制解析 current_user
用 current_user 判断用户身份
请求里的 user_id 只能作为被查询对象，不能作为身份凭证
```

---

## 用户隔离要求

所有用户数据都要隔离。

访问任何用户相关资源前，必须判断：

```text
当前用户是谁
目标资源属于谁
当前用户是否有权限访问目标资源
```

普通用户只能访问自己的数据。

如果系统有团队、组织、项目空间等概念，还必须判断：

```text
当前用户是否属于该空间
当前用户在该空间内是什么角色
该角色是否允许当前操作
```

禁止只根据资源 id 查询后直接返回。

错误方向：

```text
查到 session_id 就返回
查到 summary_id 就返回
查到 screenshot_id 就返回
```

正确方向：

```text
查到资源后继续校验 owner / team / role
没有权限就返回 403
```

---

## 团队权限要求

如果存在团队或组织，所有团队内资源必须带权限边界。

需要保护的资源包括：

```text
成员信息
邀请信息
屏幕会话
截图记录
视觉识别结果
摘要结果
团队设置
审计日志
```

基本规则：

```text
成员只能看自己的数据
管理员可以看团队内管理数据
任何人都不能访问其他团队的数据
```

如果项目已有角色规则，以项目现有规则为准，不擅自新增角色。

---

## 屏幕会话要求

开始屏幕共享时：

```text
必须已登录
必须有目标空间权限
会话必须绑定当前用户
会话必须绑定目标空间
避免同一用户重复开启多个 active 会话
```

停止屏幕共享时：

```text
必须已登录
必须校验会话归属
普通用户只能停止自己的会话
管理员是否能停止他人会话，按项目现有规则处理
停止后不应继续接收该会话的新截图
```

禁止：

```text
跨用户停止会话
跨团队停止会话
停止后继续上传截图
用前端传来的 user_id 决定会话归属
```

---

## 截图上传要求

上传截图必须校验：

```text
用户已登录
用户有目标空间权限
会话存在
会话属于当前用户或符合管理员规则
会话仍然有效
文件类型合法
文件大小合法
```

截图记录必须绑定：

```text
当前用户
目标空间
当前会话
采集时间
```

禁止：

```text
无会话上传截图
向已停止会话上传截图
跨用户上传截图
跨团队上传截图
普通接口随意返回原始截图
```

---

## 视觉识别要求

模型调用放在业务层或项目已有模型服务中。

不要把模型调用散落在接口函数里。

模型配置必须来自配置系统或环境变量：

```text
模型地址
模型名称
API key
超时时间
```

禁止硬编码真实密钥。

模型输出不要直接无校验写入关键业务结果。
至少要做：

```text
空结果处理
异常处理
超时处理
失败状态记录
```

如果输出用于摘要，尽量保存可解释的中间结果，例如：

```text
屏幕类型
可见文字摘要
活动描述
置信度
```

---

## 摘要要求

摘要必须基于已保存的识别结果生成。

生成摘要时必须带隔离边界：

```text
用户
团队 / 组织 / 项目空间
时间范围
```

查询摘要时必须重新做权限判断。

禁止：

```text
把不同用户的数据混在一起
把不同团队的数据混在一起
普通用户查看其他用户摘要
摘要接口返回原始截图
```

---

## 邀请码要求

创建邀请码时：

```text
必须已登录
必须有目标空间权限
必须符合项目已有角色规则
```

使用邀请码时：

```text
校验是否存在
校验是否过期
校验是否禁用
校验使用次数
避免重复加入
```

禁止通过邀请码绕过团队权限。

---

## 审计日志要求

关键操作应记录审计日志：

```text
创建团队 / 空间
创建邀请码
使用邀请码
修改设置
开始会话
停止会话
上传截图
生成识别结果
刷新摘要
```

审计日志不要记录：

```text
密码
token
API key
原始截图
完整敏感屏幕文本
```

---

## 配置安全要求

所有可变配置和密钥必须从配置系统或环境变量读取。

包括：

```text
数据库地址
鉴权密钥
模型 API 地址
模型 API key
模型名称
截图保存目录
截图大小限制
默认抽帧间隔
```

禁止：

```text
把真实 API key 写进代码
把真实 token 写进代码
把本机绝对路径写死
把生产配置写死
```

---

## API 合同要求

如果请求或响应结构改变，必须同步更新：

```text
后端 schema / DTO
前端 API 调用封装
相关调用点
```

不要让前端猜字段。
不要随意改字段名。
不要为了临时方便返回未约定的复杂结构。

---

## 错误处理要求

错误要清楚、稳定、不给前端泄露内部细节。

建议规则：

```text
未登录：401
无权限：403
资源不存在：404
参数错误：400
文件太大：413
模型调用失败：502 或项目已有错误格式
```

禁止直接把内部异常堆栈返回给前端。

---

## 禁止事项

不要：

```text
主动重写项目结构
主动替换数据库
主动引入队列或工作流系统
绕过鉴权
相信前端传来的 user_id
只按资源 id 查询而不校验归属
让普通用户访问他人数据
让用户跨团队访问数据
硬编码 API key
把模型调用塞满接口层
让摘要接口返回原始截图
把敏感内容写进日志
```

---

## 完成标准

后端任务完成时，必须确认：

```text
鉴权没有绕过
用户隔离没有绕过
团队 / 空间权限没有绕过
截图和摘要都有明确归属
模型密钥没有硬编码
API 合同同步更新
错误处理清楚
敏感信息没有进入日志
没有做任务外的架构迁移
```

```

这个版本就更像真正的 skill：  
不绑定文件名，不重复项目背景，只告诉模型**做后端时必须怎么做、不能怎么做**。
```
