# ScreenPulse 阶段技能拆分说明

本文档面向项目负责人，用中文说明如何把一个“大 skill”拆成多个按阶段使用的小 skill。

这里的 skill 不是具体代码文件，而是一组可复用的工作能力说明。拆分目标是让每个阶段都有明确职责、输入、输出和边界，避免一个大 skill 同时负责需求、设计、开发、测试和文档，最后难以维护。

## 一、为什么要拆分 skill

一个大 skill 如果覆盖整个项目，通常会出现几个问题：

- 职责太宽，执行时容易读太多文件。
- 容易把需求、实现、测试、文档混在一起。
- 后续调整某个阶段时，需要改动整份 skill。
- 不同阶段的验收标准不清晰。
- 多个 agent 并行时，不容易拆分写入范围。

拆分后，每个 skill 只负责一个阶段。这样可以让项目推进更稳定，也方便保留不同阶段的状态和产物。

## 二、建议拆分结构

建议把 ScreenPulse 的大 skill 拆成以下阶段：

| 阶段 | skill 名称建议 | 主要职责 |
| --- | --- | --- |
| 1 | 需求澄清 skill | 把想法整理成可验收需求 |
| 2 | 产品边界 skill | 明确 MVP 做什么、不做什么 |
| 3 | 架构定位 skill | 判断改动应落在哪一层 |
| 4 | 后端实现 skill | 修改 FastAPI、模型、服务和测试 |
| 5 | 前端实现 skill | 修改 Next.js 页面、组件和 API contract |
| 6 | AI 分析链路 skill | 维护截图识别、模型调用、摘要生成 |
| 7 | 权限与安全 skill | 检查团队隔离、角色权限、认证边界 |
| 8 | 日志与审计 skill | 维护审计日志、运行日志和问题追踪 |
| 9 | 验证验收 skill | 运行测试、构建、手工验收流程 |
| 10 | 文档同步 skill | 更新中文说明、README、PROJECT_MAP 等文档 |

这些 skill 可以独立存在，也可以由一个总入口 skill 按任务类型选择调用。

## 三、阶段一：需求澄清 skill

### 适用场景

当需求还停留在一句话、一个想法或一个问题描述时使用。

### 负责内容

- 把用户口头需求改写成清晰任务。
- 列出用户角色。
- 列出主要流程。
- 标出必须实现和暂不实现的内容。
- 形成可验收标准。

### 输入

- 用户自然语言描述。
- 当前 `需求/mvp.md`。
- 当前 `需求/已实现功能说明.md`。

### 输出

- 需求说明。
- 验收标准。
- 待确认问题。

### 不负责

- 不直接改代码。
- 不新增架构。
- 不替用户扩大需求范围。

## 四、阶段二：产品边界 skill

### 适用场景

当需求可能膨胀，或者需要判断某个功能是否属于当前 MVP 时使用。

### 负责内容

- 判断功能是否符合“保存截图和摘要，不保存完整录屏”的边界。
- 判断是否需要 LiveKit、任务队列、复杂后台等非 MVP 能力。
- 明确当前阶段应该先做的最小闭环。

### 输入

- `PROJECT_MAP.md`。
- `需求/mvp.md`。
- `需求/已实现功能说明.md`。

### 输出

- 本阶段要做的功能。
- 本阶段不做的功能。
- 未来可扩展但当前不实现的方向。

### 不负责

- 不做技术实现。
- 不把未来规划写成当前已实现能力。

## 五、阶段三：架构定位 skill

### 适用场景

当一个改动会触及前端、后端、数据库、服务或测试时使用。

### 负责内容

- 判断改动属于路由层、服务层、模型层、schema 层还是前端组件层。
- 按项目约定给出阅读顺序。
- 标出必须同步修改的文件。
- 防止把逻辑放错地方。

### 输入

- `PROJECT_MAP.md`。
- 最近目录的 `README.md`。
- 相关 contract 文件：
  - `backend/app/schemas.py`
  - `backend/app/models.py`
  - `frontend/lib/api.ts`

### 输出

- 改动落点。
- 影响范围。
- 文件清单。
- 验证方式。

### 不负责

- 不做大范围重构。
- 不引入新的架构层。

## 六、阶段四：后端实现 skill

### 适用场景

当任务需要改 FastAPI 接口、数据库模型、权限校验、截图处理、摘要逻辑或后端测试时使用。

### 负责内容

- 修改 `backend/app/routes/` 中的 HTTP 协调逻辑。
- 修改 `backend/app/services/` 中的副作用逻辑。
- 修改 `backend/app/models.py` 中的数据结构。
- 修改 `backend/app/schemas.py` 中的接口结构。
- 补充或更新 `backend/tests/test_api.py`。

### 输入

- 需求说明。
- 目标接口。
- 后端 README。
- 相关测试。

### 输出

- 后端代码改动。
- 对应测试。
- 接口变化说明。

### 不负责

- 不直接写前端页面。
- 不把服务逻辑塞进路由层。
- 不引入异步 DB、队列或插件架构，除非需求明确要求。

## 七、阶段五：前端实现 skill

### 适用场景

当任务需要改页面、组件、表单、状态展示、团队工作台行为或前端 API 调用时使用。

### 负责内容

- 保持 App Router route 文件轻量。
- 把浏览器行为放在 `frontend/components/`。
- 通过 `frontend/lib/api.ts` 访问后端。
- 维护登录、团队、共享、总结等 UI 状态。

### 输入

- 前端 README。
- `frontend/lib/api.ts`。
- 目标组件。
- 后端接口契约。

### 输出

- 前端组件改动。
- API contract 同步。
- 构建或页面验证结果。

### 不负责

- 不在组件里写临时裸 `fetch`。
- 不把业务行为放到 App Router thin route 文件中。
- 不改后端数据结构，除非任务明确跨层。

## 八、阶段六：AI 分析链路 skill

### 适用场景

当任务涉及截图保存、视觉模型调用、识别结果、小时总结或模型降级策略时使用。

### 负责内容

- 保存并校验上传截图。
- 调用 OpenAI-compatible `/chat/completions` 接口。
- 解析视觉识别结果。
- 刷新小时级总结。
- 维护无模型配置时的 fallback 行为。

### 输入

- `backend/app/services/analysis.py`。
- 模型相关环境变量。
- `FrameCapture`、`VisionResult`、`HourlySummary` 数据结构。

### 输出

- 分析链路改动。
- 模型调用和 fallback 行为说明。
- 必要测试或手工验证步骤。

### 不负责

- 不改成完整录屏分析。
- 不引入后台队列，除非需求明确。
- 不保存音频或实时媒体流。

## 九、阶段七：权限与安全 skill

### 适用场景

当任务涉及登录、团队访问、admin/member 权限、邀请码、会话归属或总结查看范围时使用。

### 负责内容

- 确认用户只能访问自己加入的团队。
- 确认 `admin` 和 `member` 权限边界。
- 检查会话、截图、总结都按团队隔离。
- 修改权限前先读 `backend/tests/test_api.py`。

### 输入

- `backend/app/dependencies.py`。
- `backend/app/routes/auth.py`。
- `backend/app/routes/teams.py`。
- `backend/app/routes/sessions.py`。
- `backend/tests/test_api.py`。

### 输出

- 权限改动。
- 回归测试。
- 风险说明。

### 不负责

- 不绕过团队隔离。
- 不为了前端方便放宽后端权限。

## 十、阶段八：日志与审计 skill

### 适用场景

当任务涉及操作记录、审计日志、运行日志、问题追踪或敏感信息处理时使用。

### 负责内容

- 记录关键团队操作。
- 保持审计事件团队归属清晰。
- 避免日志泄露敏感信息。
- 维护截图生命周期和模型调用失败的可追踪性。

### 输入

- `backend/app/services/audit.py`。
- 后端日志配置。
- 相关路由和服务。

### 输出

- 审计事件变更。
- 日志行为说明。
- 必要测试或手工验证。

### 不负责

- 不把审计日志当业务状态源。
- 不记录密码、token、完整敏感截图内容。

## 十一、阶段九：验证验收 skill

### 适用场景

当代码或文档改动完成后，需要确认结果是否可交付时使用。

### 负责内容

- 运行后端测试。
- 运行前端构建。
- 根据功能路径做手工验收。
- 汇总通过项、失败项和未验证项。

### 输入

- 改动说明。
- 测试命令。
- 目标用户流程。

### 输出

- 验证结果。
- 失败原因。
- 剩余风险。

### 不负责

- 不新增功能。
- 不把未验证内容写成已通过。

## 十二、阶段十：文档同步 skill

### 适用场景

当功能、架构、页面、接口或项目边界发生变化时使用。

### 负责内容

- 更新 `PROJECT_MAP.md` 中的架构事实。
- 更新最近目录 `README.md`。
- 更新中文需求和功能说明。
- 保持“已实现”和“计划中”分开。

### 输入

- 本次改动清单。
- 已验证代码路径。
- 现有文档。

### 输出

- 同步后的文档。
- 哪些文档未改、为什么未改。

### 不负责

- 不为了好看重写无关文档。
- 不把未来计划写进已实现功能。

## 十三、总入口 skill 的职责

如果仍然保留一个总入口 skill，它应该只做调度，不亲自负责所有细节。

总入口 skill 可以负责：

- 判断当前任务属于哪个阶段。
- 选择一个或多个小 skill。
- 给出最小执行计划。
- 确认最终验收结果。

总入口 skill 不应该负责：

- 同时读完整个仓库。
- 同时改前端、后端、测试、文档而不分范围。
- 替代所有阶段 skill 的细节规则。

## 十四、推荐使用方式

### 小需求

```text
需求澄清 skill
→ 架构定位 skill
→ 对应实现 skill
→ 验证验收 skill
```

### 跨前后端需求

```text
需求澄清 skill
→ 产品边界 skill
→ 架构定位 skill
→ 后端实现 skill
→ 前端实现 skill
→ 验证验收 skill
→ 文档同步 skill
```

### 权限相关需求

```text
需求澄清 skill
→ 权限与安全 skill
→ 后端实现 skill
→ 前端实现 skill
→ 验证验收 skill
→ 文档同步 skill
```

### AI 分析链路需求

```text
产品边界 skill
→ AI 分析链路 skill
→ 后端实现 skill
→ 验证验收 skill
→ 文档同步 skill
```

## 十五、拆分后的维护规则

- 每个 skill 只负责一个阶段。
- 每个 skill 都要写清楚输入、输出和不负责的内容。
- 涉及代码的 skill 必须有验证方式。
- 涉及文档的 skill 必须区分已实现、当前边界和未来计划。
- 当项目架构变化时，先更新架构定位 skill 和 `PROJECT_MAP.md`。
- 当产品边界变化时，先更新产品边界 skill 和中文需求文档。

## 十六、当前建议优先拆出来的四个 skill

如果不想一次拆太多，建议先拆这四个：

1. 产品边界 skill：防止 MVP 范围膨胀。
2. 架构定位 skill：防止代码放错层。
3. 权限与安全 skill：保护团队隔离和角色权限。
4. 文档同步 skill：保证中文说明和项目事实一致。

这四个最适合先拆，因为它们能覆盖当前项目最容易出错的地方。
